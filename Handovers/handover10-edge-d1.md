# Handover 10 — Edge-native Forum Feedback on D1 `forum-db`

Build target: `secure-pod-v1.7-webauthn-only` (versionCode 5), Worker
revision shipped 2026-05-22.

This handover closes out the loop H9 opened: cooperative submissions
no longer touch the local Node listener at all. The Worker verifies
the signed Ed25519 bundle, writes to a Cloudflare D1 database
(`forum-db`, `b23bdfa3-d8ba-4092-a5ac-ada4d697bc3b`), and answers the
phone directly from the edge. The local listener stays online for
private receipts, member registration, and as a best-effort mirror of
signing-key registration — but it is no longer on the critical path
for feedback ingest.

Verified working end-to-end on a phone today: feedback rows land in
`forum_feedback` with status "synced" and the only identifier on the
row is `sha256(public_key_hex)`. No PII (no email, no email hash, no
raw text from the user's address) is present on the cooperative
ledger.

User intent that prompted the move:

> the error in the app is the saved locally, but sync needs retry:
> email proof required for cooperative export. … For ease, why not use
> the Cloudflare D1 "forum-db" I used for testing.

---

## 1. Why edge-native D1

H9 dropped the zkEmail gate in the listener code, but in the field the
gate kept re-appearing because the running `forum-backend.service`
process was holding a pre-H9 copy of `listener.js`. A
`systemctl restart` fixed it, but the failure mode showed the
architectural risk: anything served behind Cloudflare Tunnel through a
self-hosted Node listener is one stale process away from regressing.

The Worker is the right place for `/api/forum/feedback` because:

- Ed25519 verification is pure Web Crypto and already lives in
  `forum-airlock/pod-signing-web.js` (used by the Personal Pod DO).
- D1 is a managed SQLite — no daemon to forget to restart.
- The phone already reaches the Worker for `/api/pod/*`; routing
  feedback through the same surface removes one tunnel hop.
- Wrangler can be redeployed from anywhere; the listener requires
  workstation access.

The listener still handles `/submit` (private encrypted receipts into
`forum-airlock/forum_inbound.db`) and `/api/register-member` because
those rows feed local analysis jobs. Migrating them to D1 is a future
step, not a blocker for the v1.7 pilot.

### Auto-created D1 schema

`ensureForumD1Schema()` runs on the first edge feedback call (and on
each `/api/register-signing-key`). It is idempotent. Tables it creates
inside `forum-db` (additive — your existing `verified_residents`,
`participation`, `responses`, `sentiment_reports`, etc. are
untouched):

```
forum_feedback     — one row per opt-in cooperative submission
forum_exports      — provenance / audit log
forum_payloads     — base64 of the signed envelope, keyed on receipt_id
edge_signing_keys  — TOFU registry: session_id -> public_key_hex
edge_replay_cache  — signature dedupe across the 5-min window
                     (PRIMARY KEY = signature for concurrency safety)
```

Columns on `forum_feedback`:

| Column | Purpose |
|--------|---------|
| `receipt_id` (PK) | Random 8-hex generated by the Pod |
| `kind` | `behavioral` / `psychographic` / `civic` |
| `category_code` | granular slug (e.g. `purchasing`, `value`) |
| `category_label` | human-readable category |
| `zip_code` | optional, set by user |
| `comment` | user-supplied text |
| `email_hash` | **misnamed for schema-compat — actually `sha256(public_key_hex)`** |
| `domain_hash` | always `NULL` after H9 |
| `web_id`, `session_id`, `public_key_hex`, `signature_hex` | bundle provenance |
| `consent_at`, `policy_version` | consent provenance |
| `encrypted_blob` | base64 of the signed payload JSON |
| `created_at` | edge ingest timestamp |
| `wiped_at` | reserved for future user-initiated wipe |

The `email_hash` column name is kept solely so the schema lines up
with the H7/H8 expectations in `forum_inbound.db`. There is no email
material in it. A future iteration can `ALTER TABLE … RENAME COLUMN
email_hash TO member_hash` once nothing downstream reads the old name.

## 3. Request flow (post-H10)

```
phone                          secure-worker                              D1 forum-db
  │                                  │                                       │
  │ POST /api/pod/civic/submissions ─►│ (forwards to PersonalPodDO)           │
  │ ◄────────────────── 200 ──────────│                                       │
  │                                  │                                       │
  │ POST /api/forum/feedback ───────►│ verifySignedBundle(bundle)            │
  │   (signed Ed25519 bundle)        │ TOFU edge_signing_keys ───────────────►│
  │                                  │ INSERT OR IGNORE edge_replay_cache ──►│
  │                                  │ INSERT forum_payloads ────────────────►│
  │                                  │ INSERT forum_exports ─────────────────►│
  │                                  │ INSERT forum_feedback ────────────────►│
  │ ◄──── 200 {"storage":"d1:forum-db"}                                       │
```

What the cooperative analyst sees per row: kind, category, zip,
comment, member_hash (= sha256 of the device pubkey), consent stamp,
signature provenance. What they cannot see: the actual email address,
the user's name, any other identifier from the device.

## 4. Operational notes

### Deploying

```bash
cd ~/Desktop/forum-airlock
npm run build:pod
npm run deploy:worker
```

The Pod bundle still ships through `dist/` so a Worker redeploy alone
won't pick up Pod UI changes — build the Pod first.

### Sanity-check after deploy

```bash
# Edge verifier should answer (not the listener):
curl -is -X POST https://secure-worker.forum-community.workers.dev/api/forum/feedback \
  -H 'Content-Type: application/json' -d '{}' | head -2
# HTTP/2 401
# {"message":"Payload authentication failed: invalid_structure"}

# Confirm the schema landed in D1:
npx wrangler d1 execute forum-db --remote --command \
  "SELECT name FROM sqlite_master WHERE type='table' AND name LIKE 'forum_%' OR name LIKE 'edge_%';"

# After a real phone submission:
npx wrangler d1 execute forum-db --remote --command \
  "SELECT receipt_id, kind, category_code, substr(email_hash, 1, 12) AS member_hash_prefix, created_at
   FROM forum_feedback ORDER BY created_at DESC LIMIT 5;"
```

### Live debugging

```bash
cd ~/Desktop/forum-airlock
npx wrangler tail secure-worker                       # all requests
npx wrangler tail secure-worker --status error        # errors only
npx wrangler tail secure-worker --search "forum_feedback"
```

Observability is still `enabled = false` in `wrangler.toml`; `tail` is
the in-session stream and works regardless.

### Rolling back

The Worker change is single-file and the listener still has the H9
ingest handler. To roll back to listener-side feedback:

1. Revert the `/api/forum/feedback` block in `secure-worker.js` to the
   pre-H10 `fetch(${listenerBase}${FORUM_FEEDBACK_PATH}, …)` proxy.
2. Revert the `[[d1_databases]]` binding back to `forum_inbound` /
   `cc896efb-245c-4015-9768-5f38337486d9`.
3. `npm run deploy:worker`.

The D1 rows already written stay where they are; nothing depends on
the binding name once data has landed.

## 5. What still routes through the listener

| Path | Why it stays |
|------|--------------|
| `/submit` / `/api/forum/receipt` | Private encrypted receipts feed `vault.py` analysis. Edge migration would require porting the vault into the Worker. |
| `/api/register-member` | Writes a row into `forum_inbound.db members` consumed by analysis CSVs. |
| `/api/register-signing-key` (mirror) | The Worker writes D1 first; a best-effort fetch to the listener keeps `forum_inbound.db pod_signing_keys` in sync for any legacy analyst tool. Listener failure does not block the device. |
| `/api/zkemail/*` | Dormant. Kept mounted so pre-v1.7 Pods still get a defined response. |

## 6. Known follow-ups (none blocking)

| Item | Notes |
|------|-------|
| Rename `email_hash` → `member_hash` | `ALTER TABLE forum_feedback RENAME COLUMN email_hash TO member_hash` once consumers stop reading it. Same for `forum_exports`. |
| Migrate `/api/forum/receipt` to D1 | Would let the cooperative go fully serverless. Blocked on porting `vault.py` (Python, Fernet) to a Worker. Workers Crypto has AES-GCM but not Fernet; either swap the encryption format or run vault server-side and accept the listener dependency. |
| Multi-device key transfer via QR | Settings already has copy-paste blob; QR (`qrcode-generator`) + optional PIN-wrapped AES-GCM is a clean follow-up. |
| Delete `public/zk-email/` README | 1.2KB of dead docs in the APK bundle. Safe to remove next iteration. |
| Wrangler v3 → v4 | Wrangler currently warns about the v3 → v4 upgrade. Non-blocking; do it during a quieter window because v4 has a few config schema renames. |
